# 1. 数据隔离约束

## 1.1 时间窗口硬约束

| 窗口 | 任务一 | 任务二 | 任务三 | 任务四 |
|---|---|---|---|---|
| W0 | 基线/初始训练 | 可用于历史基线 | 按策略使用 | 不作为最终独立准入 |
| W1-W3 | 监控；标签成熟时可计算性能 | 已成熟标签可验证 | 可训练 | 可做普通离线验证 |
| W4/OOT | 不作为任务一至三常规标签窗口 | **禁止读取标签** | **禁止训练和自动重试** | 最终独立准入专用 |

```text
硬约束：
- W4/OOT 不进入任务三训练
- W4/OOT 不参与策略自动重试
- 任务一至任务三不得读取 W4/OOT 标签
- W4/OOT 的 allows_training 和 allows_iteration_label 必须为 false
- W4/OOT 的 is_frozen 必须为 true
- allows_deployment_label 为 true（仅任务四使用）
- 任何代码路径都必须通过 DatasetAccessPolicy 拦截违规读取
- 违规访问写入 audit.data_access_violations
```

## 1.2 NATURAL 与 SCENARIO 双轨约束

```text
- NATURAL：原始时间演进和真实数据分布产生的运行事实
  → 可进入真实业务先验聚合

- SCENARIO：人为注入缺失、异常值、分布迁移、政策变化等场景
  → 独立统计，禁止直接覆盖 NATURAL 先验
  → scenario_id 必填

- KG Calibration Job 按 NATURAL/SCENARIO 分轨聚合
```

# 2. 决策边界约束

## 2.1 各模块决策职责

```text
KnowledgeService / Neo4j：
  → 输出候选路径和策略建议
  → 不生成 AUC、KS、PSI、p/q 值
  → 不执行 Validator
  → 不确定 primary_root_cause
  → 不选择 selected_strategy
  → 不执行发布、回滚或模型版本切换

DiagnosisService：
  → 执行 D/R/C/T/I，计算 Top-K Root Causes
  → 只排序 RootCause，不排序 Strategy

IterationSubgraph：
  → 排序 Strategy，训练 Challenger
  → 不重新计算 Top-K Root Causes

DeploymentService Gatekeeper：
  → 决定晋升、继续、暂停、降流、回滚或终止
  → 图谱只提供建议，不能替代决策

LLM / GraphRAG：
  → 只负责知识发现、解释和受限规划
  → 不生成数值 Evidence
  → 不自由创建未注册 Label 和关系类型
  → 不生成 AUC、KS、PSI、p/q 值或运行 Evidence
```

## 2.2 任务间决策传递约束

```text
- 任务二输出 primary_root_cause 和 recommended_action
- 任务三必须根据已确认主根因重新查询 RootCause → Strategy
- 任务三不重新计算 Top-K Root Causes
- 任务四不回到任务二重新诊断
- 任务四独立调用 KnowledgeService 查询部署知识
- 部署失败分析（DEPLOYMENT_FAILURE_ANALYSIS）创建新 lifecycle_run，
  通过 parent_lifecycle_run_id 关联原运行
```

## 2.3 recommended_action 与 need_iteration 约束

```text
recommended_action 枚举（任务二正式动作分类）：
  MODEL_ITERATION
  DATA_REPAIR
  PIPELINE_REPAIR
  THRESHOLD_ADJUSTMENT
  CONTINUE_OBSERVATION
  MANUAL_REVIEW
  NO_ACTION

派生关系：
  need_iteration = (recommended_action == MODEL_ITERATION)

数据库 CHECK 约束：
  need_iteration = true 当且仅当 recommended_action = MODEL_ITERATION

任务三入口校验：
  recommended_action 必须为 MODEL_ITERATION
  否则不得进入 IterationSubgraph
```

# 3. LangGraph State 约束

## 3.1 允许进入 State 的数据

```text
runId
版本号
当前阶段
布尔判断
计数
最高严重度
主根因摘要
最终决策
current_round
current_strategy
current_training_job_id
best_challenger_version
```

## 3.2 禁止进入 State 的数据

```text
完整 alerts
全部 Evidence
DataFrame
文档正文
训练历史列表
灰度指标时间序列
模型文件
使用 operator.add 持续追加 experiment_history 或 deployment_metrics
```

## 3.3 State 与业务数据边界

```text
- LangGraph State 仅保存流程控制字段、结果摘要和 runId
- 完整业务数据进入 PostgreSQL、MLflow 或 MinIO
- LangGraph Checkpoint 保留 State 历史，生命周期结束后按策略清理
- 正式审计数据长期保存在业务表
- LangGraph Checkpoint 不替代业务数据库
```

# 4. 任务一（监控）约束

## 4.1 Alert Context 约束

```text
- 顶层使用 monitoring_run_id 标识本次监控与告警集合
- 不设置顶层综合 alert_id
- 逐项告警使用 alert_details[].alert_id
- alert_details[].alert_id 对应 monitoring.monitoring_alerts.alert_id
- Sentinel 的 anomaly_probability 和 top_signals 只作为综合异常上下文
- 不能代替 alert_details[]
```

## 4.2 指标与告警分离约束

```text
- 所有已计算指标写入 monitoring.monitoring_metrics
- 只有触发规则的异常指标进入 monitoring.monitoring_alerts 和 alert_details[]
- 不可计算时 current_value 为 NULL，不得以 0 代替
- availability_status = NOT_APPLICABLE 时 normalized_score 为空，不能填 0
```

## 4.3 任务一职责边界

```text
- MonitoringNode 是粗粒度 LangGraph 节点
- 内部调用 MonitoringService 完成固定 Pipeline
- 不维护 Neo4j 拓扑
- 不生成根因结论
```

# 5. 任务二（诊断）约束

## 5.1 Diagnosis Context 约束

```text
- 是 KnowledgeService 返回的候选知识召回包
- 不是最终诊断结论
- 不返回 Strategy 候选
- 不负责选择任务三策略
- 不包含正式 Top-K Root Causes
- Qdrant 返回的 supporting_documents 可为空
```

## 5.2 ValidationPlan 约束

```text
- 由 DiagnosisService 根据 Diagnosis Context、数据可用性、预算和方法适用性确定性生成
- 不由 LLM 自由规划
- 反事实修复是 ValidationPlan 中产生 R 类 Evidence 的验证步骤
- 不是任务三正式训练
- 所有 required 方法必须执行
- optional 方法受预算允许
```

## 5.3 Evidence 约束

```text
- NOT_APPLICABLE 必须显式保存，不能以 normalized_score=0 代替
- applicable=false 时 normalized_score 为空，不能填 0
- evidence_score 必须来自 Validator 的确定性统计计算
- LLM 不参与数值生成
- 每条 Evidence 记录 method_code、参数、控制变量、Bootstrap 次数、p/q 值和置信区间
```

## 5.4 根因排序约束

```text
- 任务二只排序 RootCause，回答"为什么异常"
- Top-K 不包含正式策略排序结果
- PathRanker 将图谱关系权重快照与本次 Evidence 分离保存
- KnowledgeService 只负责候选路径召回，不替代 DiagnosisService 进行本次根因判定
- 权重公式进入 YAML 并保留 rule_version
```

# 6. 任务三（迭代）约束

## 6.1 入口约束

```text
- 任务三默认只围绕 primary_root_cause_code 选择策略
- 仅当主根因策略不可执行、多个根因分数接近或系统明确启用复合根因模式时，
  才读取次级根因
- 该行为必须由规则配置触发，不能由 LLM 自由扩展
```

## 6.2 策略排序约束

```text
- 任务三只排序 Strategy，回答"如何修复"
- 不重新计算 Top-K Root Causes
- 主根因分数不重复加入各策略的相对排名
- strategy_rank_score 用于候选策略之间排序
- final_strategy_confidence 用于判断是否允许自动执行
- 二者不得混用
```

## 6.3 Training Worker 约束

```text
- 技术超时：使用相同 training_job_id 幂等重试
- 模型训练成功但不达标：属于业务循环，切换策略或调整计划
- 不是技术重试
- 相同 training_job_id 重试不得创建重复实验或重复模型版本
```

## 6.4 策略配置约束

```text
- 策略选择受 YAML 中 max_iteration_rounds、allowed_strategies 限制
- 受 training_window_policy 限制
- 受 Risk Guard 限制
- 迭代震荡达到阈值后停止自动迭代
```

# 8. 知识库与 KnowledgeService 约束

## 8.1 KnowledgeService 运行边界

```text
- 所有业务模块通过 KnowledgeService 访问 Neo4j、Qdrant 和版本化规则
- 禁止在业务 Service 中直接编写 Cypher 或 Qdrant SDK 调用
- KnowledgeService 不生成 AUC、KS、PSI、p/q 值
- KnowledgeService 不执行 Validator
- KnowledgeService 不确定 primary_root_cause
- KnowledgeService 不选择 selected_strategy
- KnowledgeService 不执行发布、回滚或模型版本切换
```

## 8.2 Query Profile 约束

| Profile | 调用方 | 阈值约束 |
|---|---|---|
| exploration | 影子/研究 | min_effective_weight ≥ 0.10, min_case_count ≥ 0, min_confidence ≥ 0.00 |
| production_diagnosis | DiagnosisNode | min_effective_weight ≥ 0.60, min_case_count ≥ 10, min_natural ≥ 5, min_confidence ≥ 0.50 |
| automatic_iteration_strategy | IterationSubgraph | min_effective_weight ≥ 0.70, min_case_count ≥ 15, min_natural ≥ 8, min_confidence ≥ 0.58 |
| production_monitoring | MonitoringNode | 按监控配置阈值 |
| production_deployment_advice | DeploymentNode | 按部署治理阈值 |

```text
- exploration 关系不能绕过 production 门槛驱动自动迭代或部署
- 关系成熟度是查询时派生结果，不写成固定状态
- 低成熟度关系只能用于探索或影子验证
```

## 8.3 Neo4j 图谱约束

```text
- Neo4j 只有一张统一关系图
- 不持久化 CANDIDATE/PUBLISHED 成熟度
- 新关系从较低 initial_prior_weight 开始
- 合法关系可低权重幂等入图
- 同一 relation_key 不重复
- 每条关系可追溯来源
- 单次 Evidence 不在线写入 Neo4j
- 单次运行 Evidence 不直接修改 Neo4j effective_weight
```

## 8.4 知识抽取约束

```text
- GraphRAG/LLM 抽取结果必须包含：
  source_document_id、source_chunk_id、source_quote
  extraction_model、prompt_version、extraction_confidence
- 同步 Neo4j 前执行：
  Schema 校验、关系端点校验、稳定编码校验
  来源校验、执行器注册校验、relation_key 去重校验
- GraphRAG/LLM 不得自由创建未注册 Label 和关系类型
```

## 8.5 文档版本约束

```text
- 同一文档最多只有一个 is_active=true 版本
- 新版本全部 Chunk 同步成功后才切换 ACTIVE
- 禁止直接覆盖旧版本
- 禁止在查询期间出现半新半旧上下文
- 删除采用"先建新、再切换、后清理"顺序
```

## 8.6 Qdrant 检索约束

```text
- Qdrant 检索结果只能作为来源文本和案例参考
- 不得返回或替代：
  primary_root_cause、path_score
  strategy_rank_score、selected_strategy
  deployment_decision
- Qdrant 返回的历史案例不能直接判定本次根因
- 正式 Gatekeeper 不得依赖 Qdrant 自由文本结果做不可逆决定
```

# 9. 存储边界约束

## 9.1 各存储职责

```text
PostgreSQL：
  = 业务事实 + 原始 Evidence + 文档版本/Chunk 权威文本
  + Qdrant 同步状态 + 知识主数据 + 聚合计算 + 审计

MinIO：
  = 原始文件 + 标准化 Markdown + 表格 JSON + 报告文件

Qdrant：
  = Chunk 文本检索副本 + Dense/Sparse Vector + Payload
  = 可重建的检索投影

Neo4j：
  = 统一知识关系图的在线查询投影
  = 当前 effective_weight、案例数和置信下界

MLflow：
  = 模型实验、参数、指标、模型产物和版本
```

## 9.2 关键存储约束

```text
- PostgreSQL 是 Qdrant 的 source of truth
- Qdrant 可从 PostgreSQL + MinIO 全量重建
- Qdrant 不能成为唯一文本或审计来源
- PostgreSQL 不安装 pgvector，不保存正式向量
- PostgreSQL 不保存 Dense/Sparse Embedding
- PostgreSQL 与 Qdrant 不做请求内无补偿双写
- Qdrant 写入只由 vector_sync_outbox 驱动
- 任何运行 Evidence、权重聚合和审计数据不得只存在 Qdrant
```

## 9.3 向量同步约束

```text
- Qdrant point_id = chunk_id（固定绑定，UUID）
- 重复 Upsert 必须幂等
- 同一 Chunk 重试不会产生重复 Point
- Embedding 模型升级：新 Collection + Alias 切换
- 避免不同维度或语义空间混写
- 保留旧 Collection 作为回滚目标
- 业务代码只访问 Alias，不直接写死物理 Collection 名称
```

# 10. API 与编码约束

## 10.1 通用约束

```text
- 所有请求携带 trace_id、schema_version
- API JSON 统一使用 snake_case
- 稳定编码在模块间冻结：
  alert_code、root_cause_code、hypothesis_code、method_code、strategy_code
- 未知编码必须明确返回 INVALID_*
- 禁止静默忽略未知编码
- relation_key 一旦生成不得修改
- 实体编码变更应创建新实体和新关系，通过迁移记录保留旧关系
```

## 10.2 relation_key 格式约束

```text
格式：source_entity_code + "|" + relation_type + "|" + target_entity_code

示例：
  HIGH_FEATURE_PSI|INDICATES|feature_drift
  feature_drift|RECOMMENDS|recent_weighted_retrain
  recent_weighted_retrain|MITIGATES|feature_drift
  HIGH_DEPLOYMENT_SCORE_PSI|INDICATES|deployment_distribution_mismatch
```

## 10.3 接口引用传递约束

```text
- diagnosis_run_id、monitoring_run_id、iteration_run_id 和 deployment_id 是稳定引用
- State 传递 runId 和摘要，完整数据按 ID 查询
- 模块间通过稳定 ID 和接口合同衔接
- 避免模块间读取彼此内部表结构
```

## 10.4 接口版本约束

```text
- Alert Context：schema_version = "1.2"
- Diagnosis Context：schema_version 独立维护
- 通用响应：schema_version = "1.1"（知识图谱接口）
- ValidationPlan：rule_version 冻结
- Top-K 排名：ranking_rule_version 冻结
- Evidence 聚合：aggregation_rule_version 冻结
```

# 11. 权重与校准约束

## 11.1 权重字段约束

```text
每条动态关系至少保存：
  relation_key
  initial_prior_weight
  effective_weight
  prior_strength
  evidence_case_count
  natural_case_count / scenario_case_count
  support_count / against_count / neutral_count
  support_strength / against_strength
  confidence_lower_bound / confidence_upper_bound
  alpha_value / beta_value
  weight_version
  last_calibrated_at
  enabled
```

## 11.2 权重更新约束

```text
- 在线运行不直接修改 Neo4j 权重
- 单次 Evidence 不在线修改 Neo4j
- SUPPORT 提升 effective_weight、AGAINST 降低
- 小样本必须收缩到 initial_prior_weight
- 不得直接用 success_count / total_count 作为生产权重
- SCENARIO 独立统计，不能直接覆盖 NATURAL 权重
- weight_version 可比较、应用和回滚
- 回滚失败时保持旧版本
```

## 11.3 Observation 投影约束

```text
- 唯一约束：relation_key + source_domain + source_record_id
- 防止同一 Evidence 被重复聚合
- source_domain 取值：DIAGNOSIS / ITERATION / DEPLOYMENT
```

## 11.4 权重校准流程约束

```text
1. 任务二/三/四业务结果 → 同事务写 Outbox
2. Observation Projector → 写 kg_relation_observations
3. KG Calibration Job → 按 relation_key 与 data_track 聚合
4. 写 kg_relation_weight_snapshots
5. 创建 kg_sync_jobs（WEIGHT_UPDATE）
6. 批量更新 Neo4j effective_weight 和 weight_version
7. PostgreSQL 先提交业务事实和 Outbox
8. Neo4j 属于外部投影，不能与 PostgreSQL 使用跨库分布式事务
9. Neo4j 失败后按幂等键重试
```

# 12. Worker 与幂等约束

## 12.1 幂等键唯一约束

```text
- training_jobs.idempotency_key 唯一
- deployment_actions.idempotency_key 唯一
- document_parse_jobs.idempotency_key 唯一
- vector_sync_outbox(aggregate_id, event_type, embedding_model_code) 唯一
- knowledge_extraction_runs.idempotency_key 唯一
- kg_sync_jobs.idempotency_key 唯一
```

## 12.2 Worker 通用约束

```text
- Worker 先检查数据库状态，再执行外部副作用
- Callback 保存原始 payload hash
- 重复但内容不一致时进入冲突告警
- 相同 job_id 重试不得创建重复记录
- RUNNING Worker 定期更新 heartbeat_at
- 超时任务进入 LOST 或重试流程
```

## 12.3 事务边界约束

```text
- 业务结果表 + Outbox Event 必须在同一 PostgreSQL 事务提交
- 节点遵循"写开始日志 → 执行业务 → 在同一事务写结果和完成日志"
- 涉及外部副作用时使用 Outbox/Inbox 或明确幂等记录
```

# 13. 降级与容错约束

## 13.1 降级规则

| 故障 | 行为 |
|---|---|
| Qdrant 不可用 | Neo4j 正式路径继续；retrieval_degraded=true；文档解释降级 |
| Neo4j 不可用 | 不得用 Qdrant 文本替代正式图路径；返回可重试错误或进入人工复核 |
| 必跑 Validator 未注册 | 返回 VALIDATION_METHOD_NOT_IMPLEMENTED，不得静默跳过 |
| Production Profile 无路径 | 可显式尝试 Exploration，仅用于人工或影子，不得驱动自动迭代 |
| Embedding 服务不可用 | Outbox 保持待处理，已有索引继续服务查询 |

## 13.2 容错约束

```text
- Qdrant 降级不影响 D/R/C/T/I 确定性执行
- 只有明确要求"必须取得来源文档"的操作才阻断
- 正式 Gatekeeper 不得依赖 Qdrant 自由文本结果
- PostgreSQL 写入成功但 Neo4j 同步失败时，PostgreSQL 仍是事实源
- 同步任务使用同一幂等键重试
```

# 14. 反事实修复与迭代边界约束

```text
- 反事实修复属于任务二 R 类 Evidence 中的验证步骤
- 不是任务三正式训练 Challenger
- 反事实修复结果规范化为 R 类 Evidence
- 不单独作为平行评分输入
- 任务三正式训练 Challenger 是另一条业务流程

- data_cleaning_rule_update、threshold_adjustment 可能不产生新模型
- DiagnosisService 应先输出 recommended_action
- 只有 MODEL_ITERATION 才进入 IterationSubgraph
```

# 15. 命名与格式约束

## 15.1 命名约束

```text
- API JSON 统一使用 snake_case（知识图谱接口明确要求）
- 部署告警字段使用 camelCase（技术开发文档 §9.4 原样保留）：
  alertCode、metricCode、championValue、challengerValue
- 数据库使用 snake_case
- stable_version 从 model_deployment_state 查询
- overfitting 的稳定假设编码为 overfitting_causes_forward_validation_decay
- W4/OOT 是任务四专用独立准入窗口
```

## 15.2 JSONB 使用边界约束

```text
适合 JSONB：
  方法扩展参数、模型指标集合、图路径节点序列
  文档元数据、Worker 原始响应

不适合只存 JSONB（必须独立成列）：
  status、model_id、relation_key、metric_code
  root_cause_code、effective_weight、created_at
```

# 16. 审计与安全约束

## 16.1 审计约束

```text
- 所有结论均保留 trace_id、baseline、方法参数、随机种子、时间和版本
- Top-K 根因路径可追溯到 Alert、静态路径、ValidationMethod 和原始 Evidence
- Knowledge calibration 记录旧权重、新权重、样本范围、计算代码版本和审批人
- LLM 输入输出保存脱敏摘要和模型版本，不保存敏感原始数据
- 每个模型版本变化、稳定版本变化和流量比例变化必须记录
```

## 16.2 安全约束

```text
- Qdrant Payload 不保存客户身份证、手机号、姓名、原始交易明细
- 进入 Chunk 前执行脱敏与内容分类
- KnowledgeService 在访问 Qdrant 前完成授权
- Qdrant Payload 过滤不能替代最终权限校验
- 高风险关系的禁用、权重越级调整和回滚接口仅向 KNOWLEDGE_PUBLISHER 或 ADMIN 开放
- Alias 切换、Collection 删除和全量重建只允许管理员执行
```

# 17. 禁止事项清单

```text
1.  禁止在任务一至任务三中读取 W4/OOT 标签
2.  禁止在任务三训练窗口中使用 W4/OOT
3.  禁止 SCENARIO 数据直接覆盖 NATURAL 先验
4.  禁止 LLM 生成 AUC、KS、PSI、p/q 值或任何数值 Evidence
5.  禁止 LLM 自由创建未注册 Label 和关系类型
6.  禁止 GraphRAG/LLM 生成运行 Evidence
7.  禁止 KnowledgeService 返回正式 strategy 排名
8.  禁止 在 Diagnosis Context 中返回 Strategy 候选
9.  禁止 任务三重新计算 Top-K Root Causes
10. 禁止 任务四回到任务二重新诊断
11. 禁止 Deployment Worker 自行判断是否应该回滚
12. 禁止 用 Qdrant 自由文本替代正式图路径验证
13. 禁止 Qdrant 返回结果直接判定根因或策略
14. 禁止 在上传 API 中直接双写 Qdrant
15. 禁止 直接覆盖旧版本文档 Chunk
16. 禁止 静默忽略未知稳定编码
17. 禁止 将 NOT_APPLICABLE 当作 normalized_score=0
18. 禁止 State 保存完整列表（alerts、Evidence、DataFrame、文档正文等）
19. 禁止 使用 operator.add 持续追加 experiment_history 或 deployment_metrics
20. 禁止 在业务 Service 中直接编写 Cypher 或 Qdrant SDK 调用
21. 禁止 在 PostgreSQL 与 Neo4j 之间使用跨库分布式事务
22. 禁止 同一 Evidence 被重复聚合（relation_key + source_domain + source_record_id 唯一）
23. 禁止 同一次诊断在不同任务中重复计权
24. 禁止 删除知识实体或关系时做物理删除（应使用 enabled=false）
25. 禁止 主根因分数重复加入各策略的相对排名
26. 禁止 strategy_rank_score 与 final_strategy_confidence 混用
27. 禁止 exploration Profile 的关系驱动自动迭代或部署
28. 禁止 旧运行表名出现在代码、接口或 Seed 中
```
