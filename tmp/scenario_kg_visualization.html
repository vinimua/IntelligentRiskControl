二、单次实验七步流程

1
加载数据 — W3 Parquet + Champion模型预测 → 1485行 × 40特征
2
Clean Control — ScenarioFactory.inject(w3, {scenario:"clean_control"}) → 不破坏，作为基准组
3
注入破坏 — ScenarioFactory.inject(w3, {scenario:"covariate_drift", intensity:0.3, affected:["loan_amount_request"]}) → 30%样本加2.5σ偏移
4
跑监控 — _monitor_one(注入副本) → 得到 performance + drift + quality
5
对比分析 — ΔAUC = 0 (没降!), PSI=0.35 (漂移很大) → 结论：单特征漂移不致命
6
生成观测 — loan_amount_request → MAY_CAUSE → AUC_DROP [AGAINST] evidence_score=0.10
7
写库 — PostgreSQL kg_relation_observations → 校准后更新 Neo4j MAY_CAUSE 权重

三、12 种破坏场景

covariate_drift

单特征加2.5σ偏移 | PSI↑, AUC?

numeric_scaling_anomaly

特征乘5倍 | PSI↑↑, AUC?

missing_rate_anomaly

30%样本设为NaN | MISSING↑, AUC?

feature_staleness

特征值用旧值替换 | PSI↑, 时差相关

key_feature_failure

指定特征全设0 | 特征完全失效

multi_root_cause

同时NaN+乘倍数 | 复合效应

concept_drift

随机打乱is_bad标签 | AUC↓↓, Brier↑↑

bad_rate_shift

随机造坏样本 | BAD_RATE↑, AUC↓

fraud_pattern_shift

表面正常→改坏 | AUC↓

customer_mix_shift

高风险人群替换 | 分布整体变化

policy_selection_shift

高负债样本替换 | 选择偏置

preprocessing_mismatch

Z-score标准化 | 完全量纲变化

七、Cypher 查询根因

AUC 下降时，哪些特征最可疑？

MATCH (f:Feature)-[r:MAY_CAUSE]->(a:Alert {entity_code: "AUC_DROP"})
WHERE r.effective_weight > 0.5
RETURN f.entity_code, f.name, r.effective_weight, r.support_count
ORDER BY r.effective_weight DESC

某个特征漂移会导致什么？

MATCH (f:Feature {entity_code: "loan_amount_request"})
      -[:MONITORED_BY]->(m:Metric)-[:TRIGGERS]->(a:Alert)
OPTIONAL MATCH (f)-[r:MAY_CAUSE]->(a)
RETURN m.entity_code, a.entity_code, r.effective_weight
ORDER BY r.effective_weight DESC
